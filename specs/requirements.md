# 要件定義

## 現行スコープ

このリポジトリで現在確定しているスコープは、次の 3 つです。

1. 投票終了後の結果ページを安定して公開すること
2. 次回作業のための正史ドキュメントを整えること
3. 旧プロトタイプや企画資料を、正史と混同しない形で保持すること

## 機能要件

### RQ-001 公開ページ

- ルート `index.html` は投票受付終了を明示する
- 締切日 `2026-03-13` を明示する
- 最終結果の順位、票数、各テーマの説明を表示する
- モバイルでも可読性を保つ

### RQ-002 正史ドキュメント

- ルート文書と `specs/` で現在の正史を辿れること
- `README.md` から参照順が分かること
- `STRUCTURE_GUIDE.md` で正史と参考レーンの境界が分かること
- `DECISIONS.md` で採用済み判断が追跡できること
- `TODO.md` で現在地と次アクションが分かること

### RQ-003 資料レーンの分離

- `specs/` は承認済み仕様のみを置く
- `docs/`, `ideas/`, `proposals/`, `sessions/`, `experiments/`, `outputs/` は参考レーンとして扱う
- 参考レーンの内容は、そのまま実装指示として使わない

### RQ-004 旧投票プロトタイプの扱い

- `html_experiment/` は現時点で公開正史ではないことを明示する
- 将来再利用する場合は、テーマ構成、文言、データモデル、デプロイ方式を更新してから使う

## 非機能要件

### NF-001 ドキュメント言語

- ルート文書と `specs/*.md` は日本語で書く
- `prompts/*.md` の新規作成は、既存ルールがない限り英語を既定とする

### NF-002 運用性

- ビルド不要で、主要な公開成果物は単体 HTML として扱えること
- デプロイ方式と現在の公開物の関係が文書上で明確であること

### NF-003 AI 可読性

- 新規エージェントが 5 分以内に現在地を把握できる粒度で要点がまとまっていること

## 既知のギャップ

- `.github/workflows/pages.yml` は Firebase Secrets 前提のままで、現在の静的結果ページと完全には整合していません
- `html_experiment/index.html` は 5 テーマ構成で、ルート `index.html` の 4 テーマ最終結果と一致しません
- `docs/plans/` の最新企画メモは、まだ `specs/` に昇格していません

## 再有効化の前提条件

ライブ投票を再び有効にする場合は、少なくとも次を満たす必要があります。

1. 採用するテーマセットを再確定する
2. Firestore データモデルと UI 文言を一致させる
3. GitHub Pages デプロイで必要な Secrets 依存を見直す
4. セキュリティルールと公開方針を再確認する
