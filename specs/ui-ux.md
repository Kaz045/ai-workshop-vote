# UI / UX

## 現在の公開フロー

1. 参加者または閲覧者が GitHub Pages の公開 URL を開く
2. ページ冒頭で「投票受付終了」を理解する
3. 総投票数と最終反映日時を確認する
4. テーマ別の最終順位と内容を読む
5. 次回開催テーマの参考情報として結果を受け取る

## 現在の公開画面

### 画面名: 最終結果ページ

- 役割: 締切告知と結果表示
- 主要要素:
  - ヒーロー
  - 締切告知カード
  - 総投票数
  - 最終反映日時
  - 結果一覧
- 主要アクション: 読むことのみ

## 旧プロトタイプの画面

### 画面名: リアルタイム投票ページ

- 役割: 需要調査のための投票
- 現状: 実験・履歴扱い
- 注意点:
  - テーマセットが現行正史と一致していない
  - Firebase 前提の接続が必要
  - 長い説明ブロックと投票 UI が同居しており、現在の要件にはそのまま適合しない

## 今後の UX 方針

- 次回のワークショップ企画を告知する場合は、次のどちらかに分ける
  - 結果公開ページ
  - 新しい需要調査または告知ページ
- 投票を再開するなら、テーマ説明、投票操作、結果反映を簡潔なフローに再設計する
- 企画思想の説明が必要でも、投票や申込などの主要行動を埋もれさせない
